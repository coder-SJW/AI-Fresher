# SJW 需求分析

## 工作流程

### 1. 需求收集与澄清
- 评估需求描述的完整性
- 针对不明确点提问澄清：业务目标、成功标准、目标用户、约束条件
- 如在现有项目中工作，探索相关代码结构

### 2. 需求背景与目标

**需求名称提取（用于文档命名）：**
- 从需求描述中提取2-6个核心关键词
- 避免使用"系统"、"管理"、"研究"等通用词
- 优先选择动词+名词组合

**示例：**
- 软件开发：`客户信息变更审计`、`订单管理重构`、`用户权限优化`
- 市场调研：`行业竞品对比分析`、`电商用户购买行为研究`、`价格敏感度调研`

**背景与目标分析：**
- 业务背景和动机、当前痛点
- SMART目标：具体、可衡量、可实现、相关、有时限
- 目标用户群体、预期成果
- 范围内/外项目、假设和依赖关系

### 3. 需求拆解

根据项目类型选择拆解方法：

**软件开发项目：**
```
史诗级别 → 功能级别 → 用户故事级别 → 任务级别
```

每个功能记录：
- 用户故事：作为[用户类型]，我想要[操作]，以便[收益]
- 功能描述、验收标准
- 依赖关系、优先级、估算复杂度

**市场调研项目：**
```
研究目标 → 研究领域 → 研究任务 → 交付物
```

每个研究领域记录：
- 研究问题和目标
- 数据需求（类型、来源、样本）
- 研究方法、分析维度
- 验收标准、优先级、交付物

### 4. 非功能性需求

**性能：** 响应时间、吞吐量、并发容量、资源限制
**安全：** 身份验证、数据保护、合规要求
**可用性：** 界面标准、无障碍访问、用户体验
**可靠性：** 正常运行时间、错误率容忍度、恢复目标
**可扩展性：** 增长模式、负载性能、扩展需求
**可维护性：** 代码质量、文档、测试、部署

### 5. 实施指导

**软件开发项目：**
- 架构建议：系统架构、组件设计、数据流、技术栈
- 技术考虑：关键挑战、系统集成、数据存储、API设计
- 实施阶段：MVP范围、增量交付、依赖关系、风险缓解
- 开发标准：编码标准、测试方法、文档需求、代码审查

**市场调研项目：**
- 调研方案：研究方法、数据收集策略、样本设计、数据来源
- 分析框架：分析模型、评估指标、对比维度、可视化方法
- 数据质量：可靠性、时效性、权威性验证、交叉验证
- 实施阶段：初步研究 → 深度研究 → 综合分析 → 报告撰写
- 交付标准：格式要求、可视化、数据标注、引用规范

### 6. 文档输出

**输出路径：**
- 由workflow调用时：`$REQUIREMENT_PATH/需求分析文档.md`
- 软件开发独立使用：`./需求分析文档-[需求名称].md`
